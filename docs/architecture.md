# Architecture

This monorepo contains nine packages that together produce a Chrome Extension (Manifest V3).

## Packages

- `@chrome-ext/popup` â€” React application built with Vite.
  - Outputs: `dist/index.html`, `dist/popup.js`, `dist/assets/*`
  - The "quick action" surface: closes on outside click, no built-in persistence beyond `chrome.storage`.
- `@chrome-ext/options` â€” React application built with Vite, structurally almost identical to `popup` (duplication is intentional here - see `MISSING_SCOPES.md`).
  - Outputs: `dist/index.html`, `dist/options.js`, `dist/assets/*`
  - Opens in its own tab (`manifest.options_page`). The appropriate surface for real configuration - demonstrates reading/writing `chrome.storage.sync` via the shared `ExtensionSettings` type, unlike the popup's message-passing demo.
- `@chrome-ext/shared-types` â€” Pure TypeScript types (`ExtensionMessage`, `ExtensionResponse`, `ExtensionSettings`), no build step.
  - `types`/`main` point straight at `src/index.ts`; every consumer uses `import type`, which is erased at compile time, so there's nothing to bundle.
- `@chrome-ext/storage` â€” Typed wrapper around `chrome.storage.sync`, built with Rollup (mjs/cjs, no IIFE - it's a library dependency, never a manifest entry).
  - `getSettings()`/`setSettings(partial)`/`onSettingsChanged(callback)`, keyed to `ExtensionSettings`. Consumed by `background` (writes defaults on install) and `options` (reads/writes/reacts to changes).
  - Its `package.json` `exports` separate `"types"` (points at `src/index.ts`, always resolvable) from `"import"`/`"require"` (the compiled `dist/*.js`, needed only for the consumer's actual bundle) - otherwise consumers' independent `tsc --noEmit` would fail after a `pnpm clean` until `storage` was rebuilt.
- `@chrome-ext/content-script` â€” Page-injected script built with Rollup.
  - Outputs: `dist/content-script.js` (IIFE), plus ESM/CJS builds for reuse.
  - Runs in an isolated JS world (standard content-script sandboxing) with access to `chrome.*` APIs.
- `@chrome-ext/page-bridge` â€” Script built with Rollup, output as a single IIFE `dist/page-bridge.js`.
  - Not a content script â€” it's injected by `@chrome-ext/content-script` as a `<script src="chrome-extension://.../page-bridge.js">` tag, which runs it in the page's own JS context ("MAIN world"). This is the standard way to reach page-defined globals from an extension, and it's why the manifest must list it under `web_accessible_resources`.
- `@chrome-ext/styles` â€” SCSS + PostCSS (autoprefixer) pipeline, run via a small `tsx` build script.
  - Output: `dist/content-script.css`, declared in the manifest's `content_scripts.css`. Because the browser injects it the same privileged way it injects `content-script.js`, it isn't subject to the host page's CSP the way inline styles or JS-created `<style>` elements would be.
- `@chrome-ext/background` â€” Background service worker built with Rollup.
  - Outputs: `dist/index.mjs` and `dist/index.cjs` (the assembler renames to `background.js`).
- `@chrome-ext/extension` â€” Assembler that creates the final extension folder.
  - Script: `src/index.ts`
  - Produces: `dist/manifest.json`, `background.js`, `content-script.js`, `content-script.css`, `page-bridge.js`, `popup.html`, `options.html`, `assets/*`, `icons/*`

## Build pipeline

1. Each package builds to its own `dist/` folder
2. The assembler (`@chrome-ext/extension`) reads built outputs from `node_modules/@chrome-ext/*/dist` (pnpm workspace symlinks)
3. The assembler copies/renames assets into `packages/extension/dist/` and generates `manifest.json`
4. `@chrome-ext/background`, `@chrome-ext/content-script`, `@chrome-ext/page-bridge`, `@chrome-ext/popup`, and `@chrome-ext/options` are discovered generically (each has a `.js` main file, copied via `copyPackageFiles()`); `@chrome-ext/popup` and `@chrome-ext/options` additionally get their built `index.html`/`assets/*` copied via a shared `copyHtmlAppPackage()` helper (renamed to `popup.html`/`options.html`); `@chrome-ext/styles` (CSS) is copied via its own dedicated step, since none of these outputs are a single JS main file. `@chrome-ext/shared-types` isn't part of the assembled bundle at all - it's type-only and erased at compile time.

## Manifest

The manifest is generated by `packages/extension/src/index.ts` with sane defaults:

- Manifest V3
- `action.default_popup` â†’ `popup.html`
- `options_page` â†’ `options.html`
- `background.service_worker` â†’ `background.js`
- `content_scripts[0].js` â†’ `content-script.js`
- `content_scripts[0].css` â†’ `content-script.css`
- `web_accessible_resources[0].resources` â†’ `["page-bridge.js"]`
- Common permissions: `storage`, `activeTab`, `tabs`, `cookies`, `alarms`, `contextMenus`

Adjust matches/permissions and other settings in `src/index.ts` as needed.

## Bundlers

- Rollup for background, content-script, and page-bridge with TypeScript plugin
- Vite for popup and options (configures output filenames to `popup.js`/`options.js` etc.)
- Sass + PostCSS (autoprefixer) for styles, run via a small `tsx` script

## Monorepo tooling

- pnpm workspaces for dependency management
- Turborepo to orchestrate `build`, `dev`, `type-check`, `lint`, and `clean` across packages
